**Author:** Yoav Lutzky, Creative AI Specialist, Lightricks

## Executive Summary

### What Was Built

LTX Desktop is a functional desktop application for AI-powered video creation and editing. It runs fully locally on consumer hardware, requires no complex installations, no ComfyUI workflows, and no manual Python dependency management -- just extract and run.

Key features:
- **Image and video generation workspace**, powered by Flux Klein 4B (images) and LTX-2 Distilled / LTX-2 Dev (video). Tested on RTX 5090 with inference times of 30-60 seconds per shot at 720p.
- **Fully functional video editing module** with an NLE-style multi-track timeline (trim, split, blade, adjustment layers, subtitles, audio mixing, Premiere Pro XML import).
- **AI shot suggestion** -- a timeline-content-aware AI agent that suggests shots consistent both visually and narratively with the existing content on your timeline.
- **Video and image "takes"** -- a different approach to standard editing. Takes are nested as layers inside each clip, so you can quickly scroll between different generations. Didn't find what you like? Hit regenerate or modify your prompt.
- **Hybrid local/cloud workflow** -- generate quickly at lower resolution locally, then upscale via API at a small cost once you've committed to your final edited timeline.
- Professional video export with FFmpeg compositing, letterbox overlays, and subtitle burn-in.
- Self-contained distributable package with embedded Python (no dependencies on the target machine).

Coming soon: retake support, A2V (audio-to-video) support.

### Who Built It

A Creative AI Specialist with deep post-production expertise but zero software development background. No prior experience with TypeScript, React, Electron, Python backends, or any of the technologies used. Extensive vibe-coding experience developing other project such as the LTX Montage plugin.

### How Long It Took

Approximately 5 working days:
- ~2 days to build the generation playground
- ~1 day to build the NLE timeline editor
- ~2 days debugging, polishing, and adding advanced features

### How It Was Done

Using "vibe coding" -- an iterative, conversational approach to software development using Cursor AI. The process involves describing what you want in natural language, reviewing what the AI builds, testing it, and refining through conversation. No code was written manually. Every line was generated by the AI based on natural-language instructions informed by deep domain expertise in post-production workflows.

### Key Takeaway

When AI handles the implementation, the most valuable skill becomes knowing *what to build* -- understanding the domain, the user needs, and the workflows. My post-production background ended up being the thing that made this work, not any coding ability.

### 2. What Was Built

#### Architecture

LTX Desktop is an Electron application with three layers:

- **Frontend:** React with TypeScript, styled with Tailwind CSS. This handles the entire UI -- the home screen, generation workspace, video editor, timeline, export modal, and settings.
- **Backend:** A Python server (using the standard library's HTTP server) running locally on the user's machine. This handles AI model inference -- loading LTX-2 and Flux models, running generation, managing GPU resources and model downloads.
- **Export Engine:** FFmpeg, spawned directly from the Electron main process. Handles video compositing, letterbox overlays, subtitle burn-in, and audio mixing using a PCM buffer approach for sample-accurate sync.

The three layers communicate via HTTP (frontend to Python backend for generation) and Electron IPC (frontend to main process for file operations, export, and system integration).

#### Feature Inventory

**Generation Workspace (Gen Space)**
- Text-to-video generation with LTX-2 Distilled and LTX-2 Dev (Fast and Pro models)
- Image-to-video generation (upload an image, generate motion)
- Text-to-image generation with Flux Klein 4B
- Tested on RTX 5090: 30-60 seconds per shot at 720p
- Prompt enhancement and negative prompt support
- Configurable resolution, duration, FPS, camera motion, and seed
- Model management: automatic download, status tracking, GPU memory management

**Video Editor**
- Multi-track timeline with NLE-standard conventions (video tracks stack upward, audio tracks stack downward)
- Track priority system (higher video tracks visually override lower ones, like Premiere/Resolve)
- AI shot suggestion -- a timeline-content-aware agent that suggests shots consistent visually and narratively with existing timeline content
- Gap detection with AI-powered gap filling (generate content to fill empty sections)
- Video and image "takes" -- generations are nested as layers inside each clip, so you can quickly scroll between different takes. Hit regenerate or modify your prompt to get more options.
- Blade tool (C key), selection tool (V key), hand tool (H key), track select forward (A key)
- Trim, split, slip, and ripple edit operations
- Alt+drag to duplicate clips
- Subtitle tracks with per-clip styling (font, size, color, position, background)
- Adjustment layers as timeline assets (letterbox/pillarbox overlays with configurable aspect ratio)
- Audio track support with mute, volume control, and waveform display
- Premiere Pro XML import for bringing in existing editorial projects
- Per-clip properties: speed, reverse, horizontal/vertical flip, opacity
- Keyboard shortcuts matching industry conventions (J/K/L for playback, I/O for in/out points, etc.)
- Toggle track output (eye icon for visibility)
- Zoom to fit entire timeline
- Hybrid local/cloud workflow: generate at lower resolution locally, upscale via API once committed to your final edit

**Export Pipeline**
- Professional video export with codec selection (H.264/MP4, ProRes/MOV, WebM/VP9)
- Resolution and quality presets
- FFmpeg-native compositing (not canvas-based -- proper NLE-style render)
- Letterbox overlay burn-in from adjustment layers
- Subtitle burn-in with full styling support
- PCM buffer audio mixing (sample-accurate, the same approach real audio engines use)
- Frame rate conversion applied globally (not per-clip) to prevent sync drift

**Application**
- Project-based workflow with timeline management
- Home screen with hero video banner
- First-run setup wizard with GPU detection
- Model download manager with progress tracking
- Settings with persistent storage
- Custom app icon and branding

### 3. The Process: How It Was Done

#### Phase 1: Bootstrapping with Visual References (~Day 1)

The very first thing I did was take a screenshot of LTX Studio -- our web-based AI video platform -- and paste it into Cursor AI along with the LTX-pipeline GitHub page, with a prompt along the lines of: "Build an Electron desktop app that looks like this, and use this pipeline for inference." Giving it both the visual reference and the actual implementation documentation meant the AI understood both what the app should look like and how to wire up the AI generation backend.

This was the single most effective decision of the entire project. Instead of spending hours describing a UI in words, the AI immediately understood the layout, the component structure, the visual hierarchy, and the general interaction patterns. And instead of guessing at the inference API, it had the real pipeline code to work from. Within minutes, I had a working Electron app with a layout that resembled the screenshot and a backend that could actually generate video.

From there, I started modifying. "Move this panel to the left." "Make the generation button more prominent." "Add a settings gear icon in the top right." Each instruction was small, specific, and immediately testable.

One of the first things I always do when starting any project is set up comprehensive logging. I told the AI: "I want detailed logs for every step -- when a model loads, when a generation starts, what parameters were used, when it completes, any errors. Every significant action should be recorded." This turned out to be one of the most important decisions of the entire project. When things broke later -- and they always do -- I could open the logs, see exactly what happened step by step, and paste the relevant log output into the conversation. The AI could then diagnose the issue from the logs rather than guessing. Without logging, debugging would have been blind.

**Key insights: Start with a visual reference, not a verbal description.** The AI interprets screenshots remarkably well. **And set up logging from day one.** Logs are your debugging lifeline when you can't read the code yourself.

#### Phase 2: Building the Playground (~Days 1-2)

With the skeleton in place, I focused on the generation workflow. I described it conversationally:

- "I need a text area for the prompt, a generate button, and a video player that shows the result"
- "Add image-to-video mode where the user can upload an image"
- "Show a progress bar with the current step during generation"

The AI handled the React state management, the API calls to the Python backend, the video player integration, and the UI styling. I never thought about `useState`, `useEffect`, or component lifecycle -- I described what the user should experience, and the AI translated that into code.

The Python backend was similar. I described what the server needed to do: "Load the LTX-2 model, accept generation requests with prompt, resolution, duration, and seed parameters, stream progress updates, and save the output video." The AI built it.

The iteration cycle at this stage was fast: describe -> generate -> test -> refine. Each cycle took minutes. When something didn't work, I described the symptom ("the video player doesn't update after generation completes") and the AI diagnosed and fixed the issue.

A significant part of this phase was optimizing inference performance. I wasn't just accepting whatever the default pipeline gave me -- I was actively seeking out community hacks shared on Discord, optimization techniques from GitHub discussions, and various implementation tricks to improve inference times and visual quality. I'd paste these into Cursor and say "implement this optimization." Then there was a lot of back and forth testing: does this actually improve inference time? Does it affect visual quality? Some worked, some didn't, some improved speed but degraded results. This empirical, iterative approach to performance tuning was only possible because I could test each change immediately without understanding the underlying code.

#### Phase 3: Building the Timeline Editor (~Day 3)

This is where my post-production background became the critical differentiator. An AI can generate code for a timeline UI, but it doesn't inherently know the conventions that make an NLE feel professional.

I started by describing the fundamental structure: "I need a multi-track timeline like Premiere Pro. Video tracks should stack upward from a divider line, audio tracks should stack downward. Each track has clips that can be dragged, trimmed, and split."

Then I layered in specific NLE conventions that only someone with editing experience would know to ask for:

- "Track priority: if a clip on V2 overlaps a clip on V1, the V2 clip should be what's visible in the preview. This is how every NLE works."
- "The blade tool should be activated with the C key. When active, clicking on a clip splits it at that point."
- "I need J/K/L playback controls. J plays backward, K stops, L plays forward. Pressing L twice does 2x speed."
- "When I import a Premiere Pro XML, it needs to preserve the track layout, clip timing, in/out points, and speed changes."
- "Adjustment layers should work like After Effects -- they're assets you drag onto a video track, and they apply effects (like letterbox) to everything below them."

I also brought in concepts from professional workflows that I'd been thinking about in the context of AI:

- **Offline/online workflow for AI upscaling:** In traditional post-production, you edit with low-res proxies (offline) and then reconnect to high-res media (online) for finishing. I applied this concept to AI upscaling -- edit with the generated clips, then upscale the ones you want to keep.
- **Gap generation:** Empty sections on the timeline (gaps between clips) could be filled by AI. The system is context-aware -- it looks at the last frame of the previous shot and the first frame of the next shot to help the user create a prompt that maintains visual and narrative continuity. Select a gap, choose text-to-video or image-to-video, and the AI generates content that fits exactly into that gap, bridging what comes before and after.

**Key insight: Domain expertise is the irreplaceable ingredient.** The AI is a phenomenal executor, but it needs someone who deeply understands the problem domain to direct it. The combination of "knows exactly what to build" + "can build anything described" is extraordinarily powerful.

#### Phase 4: Advanced Features and Integration (~Days 3-4)

With the core editor working, I pushed into more sophisticated features:

**Premiere Pro XML Import:**
This was important because it allowed me to test the timeline with real editorial content. I could edit a sequence in Premiere, export the XML, import it into LTX Desktop, and verify that clips, timing, tracks, and effects translated correctly. This immediately exposed edge cases I wouldn't have found otherwise -- clips with speed changes, trimmed media, nested sequences.

I progressively asked for more XML features: "Import should also include audio mute/volume, horizontal/vertical flips, reverse playback, and time speed/duration changes."

**Subtitle System:**
"I need a subtitle track that sits above all video tracks. Subtitles should have configurable font, size, color, and position. They should appear in the preview at the correct time."

**Export Pipeline:**
This was a technically challenging part. The first approach (rendering frames on a canvas in JavaScript and sending them to FFmpeg) was painfully slow. I told the AI: "This is very slow. Is there a more efficient way?" We went through several iterations:

1. Canvas-based frame-by-frame export (too slow -- thousands of HTTP requests)
2. IPC-based binary transfer (hung at 0% -- overcomplicated)
3. Native FFmpeg compositing from the Electron main process (correct approach)

At one point I got frustrated and said: "Are you overcomplicating it? What is the best approach?" That pushback led the AI to scrap the frame-by-frame approach entirely and instead construct FFmpeg filter graphs that do all the compositing natively.

**Audio Sync Saga:**
The most instructive debugging experience was audio synchronization. After getting video export working, the audio was completely out of sync. I described what I heard: "The audio is delayed." The AI tried fixes with FFmpeg's `adelay` and `amix` filters. It didn't work. The audio was still off. I said: "It's totally not in sync again. Why is our approach not working? HOW DO OTHER NLEs DO IT?!"

That was a separate breakthrough moment. The AI explained that real NLEs don't use FFmpeg filter graphs for audio mixing -- they do it at the raw PCM sample level. Extract each clip's audio as raw bytes, calculate the exact byte offset for its timeline position, and sum the samples. No `adelay`, no `amix`, no filter graph timing drift. Just math.

We also discovered that an FPS mismatch (25fps source from Premiere, 24fps export) was causing per-clip duration quantization that accumulated across clips. The fix: apply frame rate conversion once to the entire output (like real NLEs do), not per-clip.

**Key insight: Debugging is a conversation, and knowing when to challenge the AI's approach is crucial.** The AI will sometimes go deep into a complex solution when a fundamentally different approach is needed. As the human in the loop, recognizing "this feels too complicated" is a legitimate and valuable signal.

#### Phase 5: Packaging and Distribution (~Day 5)

The final step was making the app distributable. I needed someone on another computer to be able to run it without installing Python, PyTorch, or any dependencies.

The approach: bundle an embedded Python 3.11 environment with all dependencies pre-installed (PyTorch, FastAPI, Diffusers, LTX inference libraries) inside the Electron app package. The build system downloads Python's embeddable distribution, installs ~5GB of packages into it, and electron-builder bundles everything into a self-contained ZIP.

I was emphatic about one thing: "The installer needs to install everything in a virtual environment, otherwise it overrides all your settings on the computer." The embedded Python approach satisfies this -- it's completely isolated from the target system's Python.

The final package: a 2.88GB ZIP file. Extract it, run `LTX Desktop.exe`, and everything works. AI models (~43GB) download automatically on first use.

### 4. The Methodology: Vibe Coding Principles That Worked

After five days of intensive vibe coding, these are the patterns I noticed working well for me:

**1. Start visual, not verbal.**
Pasting a screenshot of an existing UI and saying "build this" is 10x more effective than describing a UI in words. The AI parses layouts, component hierarchies, and design patterns from images extremely well. Use screenshots of competitor products, mockups, or even hand-drawn sketches as starting points.

**2. Leverage domain expertise relentlessly.**
The AI knows how to code. You know how the product should work. This asymmetry is the key to the entire approach. I never told the AI *how* to implement track priority or blade tool logic. I told it *what* those features should do, referencing industry-standard behavior. "Make it work like Premiere" is a perfectly valid instruction when you know exactly what that means.

**3. Iterate in small, testable loops.**
Every instruction was small enough to test immediately. "Add a mute button to each track" -- test it. "When I click the mute button, the audio for that track should not play" -- test it. Never batch multiple features into one request. Small loops mean small bugs, and small bugs are easy to describe and fix.

**4. Be specific about behavior, not implementation.**
I never said "create a useState hook for the blade tool mode and add an onClick handler that calculates the split point from the mouse position relative to the timeline." I said "when the blade tool is active and I click on a clip, it should split into two clips at that point." The AI handles implementation; you handle intent.

**5. Use real-world data to test.**
Importing actual Premiere Pro XML timelines was one of the best decisions I made. Real editorial content exposed issues that synthetic test data never would -- clips with unusual trim points, speed changes, nested tracks, mixed frame rates. Always test with real data from real workflows.

**6. Debug by describing symptoms.**
"The audio is delayed by about half a second." "The first clip shows black for a moment before playing." "The letterbox isn't visible in the exported video." Describe what you observe, not what you think the code should do. The AI is better at diagnosing code issues than you are -- your job is to be a precise observer.

**7. Push back when it feels wrong.**
The AI will sometimes overcomplicate things. If a solution feels overly complex, say so. "Are you overcomplicating this?" is a legitimate prompt. Some of the best architectural decisions in this project came from me pushing back on an approach that felt wrong, even though I couldn't articulate the technical reason. Intuition matters.

**8. Set up logging from day one.**
Before building any features, make sure every significant action is logged -- model loading, API calls, generation parameters, errors, timing. When something breaks (and it will), you can paste the log output into the conversation and the AI can diagnose from evidence rather than guessing. Logs are the single most valuable debugging tool when you can't read the code yourself.

**9. You are the product manager, designer, and QA tester.**
In this workflow, the human is not the developer. You are three roles simultaneously: the PM (deciding what to build and why), the designer (deciding how it should look and feel), and the QA tester (finding bugs and verifying behavior). The AI is the engineer. Embrace these roles fully.

### 5. Lessons Learned

**What worked well:**
- The screenshot-to-skeleton approach for bootstrapping
- Setting up comprehensive logging from the very start -- this made every debugging session 10x faster because I could share exact log output with the AI instead of vaguely describing symptoms
- Using post-production expertise to drive NLE-specific features
- Small iteration loops with immediate testing
- Importing real Premiere XML to validate the timeline
- Pushing back on overly complex solutions
- The AI's ability to handle complex state management, FFmpeg command construction, and Electron IPC without me understanding any of it

**What was frustrating:**
- Debugging cycles where the AI would fix one issue but introduce another (especially with audio sync)
- The AI sometimes going in circles -- trying variations of the same broken approach rather than stepping back to reconsider fundamentally
- Needing to say "stop, this approach is fundamentally wrong" rather than letting the AI continue iterating on a bad solution
- TypeScript compilation errors from pre-existing issues that blocked builds

**Where domain expertise was essential:**
- NLE track conventions (stacking order, priority, tool behaviors)
- Understanding what "in sync" means for audio/video
- Knowing that FPS mismatch causes cumulative drift
- Offline/online workflow concepts
- What a professional export pipeline needs (letterbox, subtitles, codec options)
- Understanding that adjustment layers are assets, not track types
- Knowing what to test (import a real Premiere XML, not synthetic data)

**Where the AI led:**
- All implementation details (React state management, component architecture, CSS layout)
- Python backend structure and API design
- FFmpeg filter graph construction
- Electron IPC communication patterns
- Build system and packaging configuration
- Debugging code-level issues (null references, timing bugs, race conditions)

**Approximate time distribution:**
- ~20% describing features and giving direction
- ~10% reviewing generated code (mostly visual review of the running app, not reading code)
- ~40% testing and identifying issues
- ~30% debugging conversations (describing problems, reviewing fixes)

### 6. Where I Think This Approach Fits

Based on this experience, here's where I think vibe coding worked well and where I could see it being useful:

- **Prototyping and proof-of-concept.** Going from idea to working app in days means you can test whether something is worth building before committing a team to it.
- **Domain-specific tools.** The value here came from me knowing exactly how an NLE should behave. The AI handled the code; the domain expertise shaped the product. This probably applies to any field where someone deeply understands the problem space.
- **Internal tools and utilities.** Things that need to work but don't need to be production-grade infrastructure.

A natural next step from here would be to have engineering review the prototype -- decide whether to iterate on it, refactor it, or treat it as a living spec. The working app communicates intent much more concretely than a written document could.

I don't have illusions about the limitations. There are things the AI struggled with (it went in circles on audio sync for a while before I pushed it to rethink the approach), and the generated codebase would likely benefit from an engineering review for long-term maintainability. But as a way to go from zero to a working, testable product quickly -- it was remarkably effective.
